> Candy Box 2's quest/combat engine is a tightly coupled but elegantly simple composition of five interacting classes: Quest (scene manager), QuestEntity (game object base), CollisionBoxCollection (AABB collision), QuestEntityWeapon (contact-damage component), and QuestEntityMovement (tick-rate movement). There is no data-driven quest format — every quest is a handcrafted TypeScript subclass of Quest. Enemies are QuestEntity subclasses with their AI coded directly in update(). Combat is purely proximity-based AABB: weapons define a collision region larger than the entity body, and any opposing-team entity that intersects it takes damage on a tick delay. Gravity is a per-entity boolean flag on QuestEntityMovement, and the engine's only affordance for zero-gravity is a global Quest flag plus a per-entity antiGravity duration timer. The sea quest (TheSea) demonstrates a pattern/level wave system and global scrolling, which is the most sophisticated generalization in the CB2 codebase. CB3 must cleanly generalize gravity direction, movement physics (Newtonian thrust vs. walked), the win-condition trigger, and the wave/spawner system into a shared engine while keeping quest-specific content declarative.

### Key findings
- Quest.ts is the base scene class (937 lines). It owns: a flat QuestEntity[] array (all entities, including player, walls, enemies, spells), a RenderArea, player-spell/potion UI buttons (created from save-flags in constructor), two shared cooldown timers (spellsCountdown, potionsCountdown), global gravity/worms-like disable flags, a globalDrawingOffset (used for scrolling the whole scene), and collision walls around the quest border.
- There is NO data-driven quest authoring. Every quest is a TypeScript class extending Quest. Config is constructor code: resizeQuest(w,h), addPlayerCollisionBoxes(), getGame().getPlayer().setGlobalPosition(), hardcoded entity spawns. The Arena module uses a registry (TheArenaModule.addQuest) to attach factory functions, but actual quests are handcoded classes.
- QuestEntity is the universal game object. Its update() sequence is: handleAnimation() → handleGravity() OR handleMovement() → handleCombat(). Gravity and movement are mutually exclusive per-tick: if the entity falls, it cannot also translate horizontally that tick (gravity takes priority and returns true to skip movement).
- Collision detection is brute-force AABB O(n²): QuestEntityWeapon.handleCombat() iterates all quest entities, checks team inequality, then calls CollisionBoxCollection.collidesWith(). Each CollisionBox is position+size relative to entity globalPosition. No spatial partitioning exists.
- Auto-attack-on-contact is implemented by QuestEntityWeapon: each entity has 0-N weapons, each weapon has its own CollisionBoxCollection (often slightly larger than the entity body, e.g. Pos(-1,-1), Size(+2,+2)). The weapon checks collision every tick subject to a QuestEntityWeaponDelay (FIXED / BETWEEN / ONCE_THEN_WAIT). No explicit 'melee range' concept — collision IS the attack.
- Gravity is implemented in QuestEntity.handleGravity(): if questEntityMovement.getGravity() == true AND quest.gravityDisabled == false AND antiGravity == false, the entity tries move(0,+1) every tick. If it succeeds, it is 'falling' and the movement phase is skipped (return true). If it fails, gravity had no effect and movement runs. Jump is a duration counter (jumpDuration ticks) that moves -jumpSpeed per tick, then sets controlledFalling=true.
- Worms-like movement (QuestEntityMovement.wormsLike) allows an entity to automatically step up 1-char ledges and not fall off 1-char drops — exactly like Worms (Team17). Player uses this in most side-scrolling quests. Disabled for the sea quest.
- TheSea quest demonstrates the engine's scrolling trick: globalScrolling() calls forceMovingAllEntities(new Pos(scrollingXOffset, 0)) every tick to shift all entities left, then regenerates content at the right edge. This is how the 'world moves past the player' illusion works — the player stays near x=0, everything else shifts.
- TheSea also shows the only wave/pattern system in CB2: a TheSeaPatternLevel chain (Level0→Boss0→Level1→Boss1→...) where each level produces TheSeaPattern objects that spawn enemies in a range (x1,x2). Levels advance when isLevelDone() returns true. This is the closest CB2 has to a wave engine.
- Boss phase transitions are implemented as enum state machines on the boss QuestEntity subclass itself (GiantNougatMonsterStep: ASLEEP→AWAKE→ANGRY, triggered by HP thresholds). The boss calls addWeapon() and reDrawAscii() on phase change. Multiple bosses can coexist in one quest (Hell has both Camazotz and Devil).
- Player spells and potions are created in Quest.createPlayerSpells() by reading Saving (save-state) flags. This means spell availability is always global — there is no per-quest spell configuration. The shared countdown (spellsCountdown / potionsCountdown) gates ALL spells of their type simultaneously.
- Death handling: player.shouldDie() is checked each tick in the quest's update() method. On true: endQuest(false) is called, which calls super.endQuest(false) which sets questEnded=true. No respawn in CB2 — the player is returned to the map. Death messages come from QuestEntity.getDeathMessage() using the lastDamageReason.
- Drops (candies, items) are accumulated in Quest.candiesFound and Quest.itemsFound during the quest, then transferred to the game in endQuest(true). Items are registered via foundGridOrEqItem(new QuestItemFound(quest, savingName, message)) inside the enemy's willDie() override.
- The GalacticWars mini-game is NOT part of the Quest engine — it is a completely separate ThirdHouseGame class with its own y-scrolling ship logic, manual asteroid tracking via Pos[], and no entity system. This is the only 'space' content in CB2, and it is a throwaway mini-game, not a template.
- Teams are an enum: PLAYER, MOBS, NATURE. Weapons only deal damage to entities of a different team. The Spikes entity uses NATURE team so it hurts PLAYER-team entities but not MOBS. There is no friendly-fire concept.
- The render pipeline is: preDraw() (clear) → optional background ASCII array → drawEntities() (each entity draws itself + health bar) → drawAroundQuest() (spells UI + quest log) → addExitQuestButton() → postDraw() (calls game.updatePlace() which triggers jQuery DOM write). Called at 10 FPS via setInterval.
- QuestEntitySpell extends QuestEntity and adds a color layer system (QuestEntitySpellColor[]). Projectiles (Fireball, Blackhole) are QuestEntitySpell instances. Fireball has four target modes: NO_TARGET (dumb movement), TARGET_ENTITY (homing), TARGET_POSITION, TARGET_STICK_ON_ENTITY (follows another entity).
- The Arena module is the closest thing to data-driven quests: TheArenaModule.addQuest(new TheArenaModuleQuest('questId', factoryFn)) lets quests self-register. This is purely a string-keyed factory registry, not a config object — the factory still constructs a handcoded class.

### Patterns to steal
- The QuestEntity base class composition pattern: one class owns renderArea + cbc + movement + weapons + animation + team. Components are assigned in the constructor, not through an ECS registry. For CB3's scale this is simpler and faster than a full ECS.
- QuestEntityWeapon as a first-class object: the weapon IS the hitbox + delay + damage number. Separating weapon from entity allows enemies to have multiple weapons (e.g., the Devil has claws + fireballs), swap weapons on phase change, and makes player weapons readable from equipment config.
- The QuestEntityWeaponDelay enum (FIXED/BETWEEN/ONCE_THEN_WAIT) covers all CB2's timing needs elegantly. BETWEEN gives humanized random attack timing; ONCE_THEN_WAIT is perfect for projectile spells that need a cooldown.
- forceMovingAllEntities(offset) for global scrolling — simple and correct. Avoids a camera matrix transform and keeps all entity positions in world-space, which simplifies collision queries.
- TheSea's Pattern/PatternLevel chain: a linked list of wave 'chapters', each a self-contained spawner object that reports isPatternDone(). This is the right abstraction for authored encounter sequences.
- The willDie() / shouldDie() separation: shouldDie() returns bool (pure query), willDie() fires side effects (log messages, drops). In CB3 willDie() should also trigger death-message selection and respawn scheduling.
- The QuestEntitySpell subclass with a color layer system: CSS/DOM coloring of specific screen cells as an overlay is exactly how CB3 should do visual effects (glows, fire, void effects) given the pure ASCII + CSS-only constraint.
- Transparency via RenderTransparency: character-level masking so large ASCII art entities (shark, devil) skip drawing their 'space' characters, letting the background show through. CB3 needs this for all multi-character entities.
- The globalDrawingOffset mechanism: a single Vec2 applied to all entity draw calls, enabling scroll, screen-shake (small oscillating offset), and viewport effects without modifying any entity positions.
- The configPlayerOrClone(entity) pattern: a single method sets up both the player and any clones, ensuring they always have identical movement/weapon config. In CB3 this should be part of the QuestDef as a playerConfig object.

### Pitfalls
- CB2 mutates Pos objects in place extensively (this.globalPosition = pos, pos.add(...)). Every position operation creates aliasing bugs when the same Pos is shared. CB3 must use immutable Vec2 (return new Vec2(x+dx, y+dy)) throughout.
- The single shared spell/potion countdown means casting any spell locks out all spells. CB3 must use per-spell cooldowns or this becomes infuriating with 5+ spells.
- The O(n²) collision loop iterates ALL entities including walls and spells. CB2's entity counts stay low (<50). CB3's zero-G asteroid field can easily hit 100+ entities (50 asteroids × 2 split generations). Add spatial bucketing for mode C quests before this becomes a problem.
- QuestEntity.handleGravity() returns true to block movement in the same tick. This means an entity cannot move horizontally while falling — it falls straight down. In CB2 this is fine (enemies don't need to dodge while falling). In CB3's vertical climb and zero-G modes, this assumption breaks entirely and must be removed.
- CB2's wave spawning is fully imperative — timers and counters scattered across the quest class. There is no way to serialize 'quest progress mid-run.' CB3 should keep all wave state in a WaveScheduler object so it could theoretically be saved.
- Gravity is a boolean flag on QuestEntityMovement, not a vector. Adding 'gusts push up' (mode B) or 'inverted gravity zones' requires hacking the flag per-frame, which is fragile. Use a Vec2 gravityVector from the start.
- The QuestEntitySpellCaster (enemy spellcasting) shares the same QuestEntityWeaponDelay infrastructure but is wired through a CallbackCollection, making it hard to inspect or serialize. In CB3, enemy abilities should be expressed as typed AbilityDef objects, not raw callbacks.
- CB2 quest classes hold mutable state as class fields (currentBirdTime, mosquitoTimer, etc.) mixed with the entity-update logic. This makes per-quest state hard to snapshot and creates ordering dependencies. CB3 should keep per-quest state in a plain object that the WaveScheduler owns.
- The `addEntity()` method returns false if the entity collides on spawn, but most quest code ignores this return value. Silent spawn failures can create confusing behavior (TheSea addBigShark checks it; most quest constructors don't). CB3 should always handle spawn failure explicitly.
- CB2's `update()` method on Quest is abstract/empty in the base class but is called via the questCallbackCollection (10Hz interval). Each quest subclass implements its own update() loop. This works but duplicates the check-ended/check-dead/updateEntities/draw sequence in every quest. CB3 should push this boilerplate into the base class with hooks (onUpdate(), onWin(), onDeath()).

### Recommendations
- Introduce a QuestMode enum (HORIZONTAL, VERTICAL, ZERO_G, SHIP_BROADSIDE) and a PhysicsDriver interface. Each mode provides: gravityVector: Vec2, playerControlScheme: string, and an update(entity, dt) method. The Quest base class calls physicsDriver.update() instead of the hardcoded gravity logic. This allows all four CB3 modes to share the entity/weapon/collision system.
- Implement a data-driven QuestDef object (TypeScript literal) that Quest reads in its constructor. Mandatory fields: id, size, physicsMode, playerStart, boundaryCollisions, waves[], onWin, deathMessage. Each wave entry has: spawnFn | entityDef[], spawnTrigger (distance | timer | bossHpBelow | manual), and maxCount. This replaces the 13 handcoded quest subclasses with 13 QuestDef objects plus a handful of shared helper subclasses.
- Keep the QuestEntity/weapon/collision model unchanged — it is the right abstraction. Do convert it to immutable-style by making all position updates return new Pos instances (to comply with CB3 coding style) rather than mutating in place as CB2 does.
- Generalize the scrolling system from TheSea into a ScrollingDriver interface. HORIZONTAL quests scroll x; VERTICAL quests (beanstalk climb) scroll y; ZERO_G quests scroll both based on player velocity. The globalDrawingOffset mechanism is correct — extract it into the driver.
- For mode B (vertical climb): flip gravity to +x (rightward = down in rotated space) or simply make gravityVector a Vec2 in the physics driver. Gusts become periodic forceMovingAllEntities(new Pos(0, -gustStrength)) pulses. Gravity-inversion zones are trigger-volume entities that temporarily flip the driver's gravityVector sign.
- For mode C (zero-G drift): set gravityVector = Vec2(0,0). Add a PlayerVelocity component (Vec2 that persists between ticks). The 'gumball cannon' action adds an impulse of (-thrustMagnitude, 0) relative to facing, then the velocity decays at a configurable drag coefficient per tick. Asteroids are QuestEntity subclasses whose willDie() spawns two smaller asteroids (fractal split, max 2 generations). Running out of ammo triggers the 'drift forever' end condition instead of player death.
- For mode D (ship broadside): implement as a separate PhysicsDriver that runs a timing mini-game (broadside phase). When boarding triggers, switch the quest's PhysicsDriver to HORIZONTAL mode in place — the same entity array continues, but a new player-entity is spawned at the ship deck position. This in-quest mode switch is the key architectural choice that avoids two separate systems.
- Replace the global single-spell-countdown with per-spell cooldowns stored on each QuestPlayerSpell instance. CB2's shared countdown was a simplification that prevents using different spells at different rates. CB3 needs e.g. the gumball cannon (fast) and the sticky-ground spell (slow) to have independent cooldowns.
- Extract wave spawning from per-quest constructor code into a WaveScheduler that the QuestDef configures. WaveScheduler should support: distance-triggered waves (for scrolling quests), timer-triggered waves, event-triggered waves (bossHpBelow threshold), and manual waves (designer calls spawnWave('waveId')). This unifies the Forest's monster-handling timers, TheSea's pattern system, and the Camazotz-dropping-demons pattern.
- Implement a PhaseStateMachine component for boss entities. Each boss definition supplies an array of BossPhase objects with: hpThreshold, ascii, weapons[], behaviorFn. The state machine transitions automatically when HP crosses a threshold, calling reDraw() and swapping weapons — replacing the ad-hoc switch statements in GiantNougatMonster and Devil.
- Death/respawn: CB3 DESIGN.md says 'respawn at last safe zone, lose nothing.' Implement a SafeZone entity type that records the last checkpoint position. On player death, call quest.respawnPlayer(lastSafeZone) instead of endQuest(false). The bespoke death message pool should be in the QuestDef (deathMessages: string[]) picked randomly.
- The QuestEntityDamageReason system is good — keep it. Extend WhatType with: HAZARD (lava, spikes, vacuum), COLLISION (asteroid impact), THRUST_BACKFIRE (zero-G cannon self-damage on wall collision). This gives richer death messages for CB3's new modes.
- Move spell availability configuration from global save-flags into the QuestDef (allowedSpells: string[], allowedPotions: string[]). Some CB3 quests should restrict spells (e.g., the mint labyrinth disables teleport; zero-G quests disable jump).
- The Arena module's self-registration pattern (TheArenaModule.addQuest()) is worth keeping for the arena/sandbox mode, but CB3's main quest flow should use a central QuestRegistry that maps quest IDs to QuestDef objects. The registry validates defs at startup.
- Performance: CB2's O(n²) collision is fine up to ~50 entities. CB3's zero-G asteroid field may have 50-100 asteroids. Add a simple spatial grid (10×10 buckets) as an optional optimization that the PhysicsDriver can enable. For horizontal/vertical quests, keep the brute-force loop.

Entity Update Sequence (per tick, per entity)\n\n```\nQuestEntity.update():\n  1. handleAnimation()         → cycle ASCII frame if due\n  2. handleGravity()           → if gravity enabled AND not antiGravity AND not quest.gravityDisabled:\n                                   try move(0, +1)\n                                   if succeeds → return true (skip movement)\n  3. handleMovement()          → if NOT gravity-blocked AND NOT stopped:\n                                   movement.update()\n                                   if shouldMoveAtThisFrame():\n                                     if wormsLike: moveWormsLike(offset)\n                                     else: move(offset)\n  4. handleCombat()            → for each weapon: weapon.handleCombat()\n                                  for each spellCaster: spellCaster.tryToCast()\n```\n\n---\n\n## 4. Collision System\n\nABB brute force:\n\n```\nQuestEntityWeapon.handleCombat():\n  if delay.tryToAttack():\n    for entity in quest.entities:\n      if entity != owner AND entity.team != owner.team:\n        if weapon.cbc.collidesWith(entity.cbc):\n          owner.hit(entity, damage, damageReason)\n          delay.theWeaponAttacked()\n\nCollisionBoxCollection.collidesWith(other, offset):\n  for each box in self:\n    for each box in other:\n      AABB test using entity.globalPosition + box.position + pos offsets\n      return true on first overlap\n```\n\nKey detail: the weapon's CBC position offsets are relative to the **owning entity's globalPosition**, not the weapon's own position. So a weapon with `CollisionBox(this, Pos(-1,-1), Pos(width+2, height+2))` extends 1 cell beyond the entity in all directions — this is how 'contact damage' works.\n\n---\n\n## 5. Gravity, Jump, Anti-Gravity\n\nGravity is attempted once per tick as `move(0,+1)`. It is blocked by:\n- `quest.gravityDisabled == true` (whole-quest flag)\n- `entity.antiGravity == true` (duration-based timer)\n- `this.jumping == true` (during jump arc)\n\nJump is a separate counter:\n```\njump(duration, speed=1):\n  check: not jumping AND not controlledFalling AND checkCollision(0,+1)\n  → sets jumping=true, jumpDuration, jumpSpeed\n  → each tick: move(0,-jumpSpeed), decrement jumpDuration\n  → when done: jumping=false, controlledFalling=true\n  → controlledFalling ends when gravity move(0,+1) fails (hit floor)\n```\n\n`controlledFalling = true` means the entity still runs movement while falling (player can walk mid-air after a jump). This is the CB2 version of coyote time.\n\n---\n\n## 6. Worms-Like Movement\n\nCalled when `questEntityMovement.wormsLike == true`:\n\n```\nmoveWormsLike(pos):\n  if can move normally (no collision at pos):\n    if checkCollision(pos + (0,2)) AND !checkCollision(pos + (0,1)) AND !jumping AND !falling:\n      move(pos + (0,1))   // step down 1 char\n    else:\n      move(pos)           // normal move\n  else if checkCollision(pos + (0,-1)) is clear AND !jumping AND !falling:\n    move(pos + (0,-1))    // step up 1 char\n  // else: blocked, don't move\n```\n\nThis gives fluid platformer movement over single-char steps without needing explicit step detection.\n\n---\n\n## 7. Player Spells Architecture\n\n`createPlayerSpells()` in Quest constructor reads Saving flags (item possession) and builds `QuestPlayerSpell[]`. Each spell has:\n- A button rendered above the quest at a fixed (x,y) offset\n- A countdown type: SPELLS | POTIONS | ITEM_CAPACITIES | BLACKHOLE\n- A countdownTime that is added to the shared quest-level counter on cast\n- Optionally a `numberIdWichLimitsQuantity` save-key (potion stock)\n\n`lowerCountdowns()` decrements the shared counters each tick. Mode-Specific Notes\n\n### Mode A — Horizontal Side-Scroller\nDirect CB2 port. `wormsLike=true` for most quests. `gravityVector=(0,+1)`. Win = player.x >= questWidth.\n\n### Mode B — Vertical Climb (Beanstalk)\nRotated: player walks vertically (upward), gravity still pulls down (+y). Use the same PhysicsDriver as Mode A but the player's `configPlayerOrClone` sets movement to `(0,-1)` (upward). Gusts: periodic `forceMovingAllEntities(0, +gustStrength)` pulses. Gravity-inversion zones: `QuestEntityTriggerVolume` entity that sets `driver.gravityVector = Vec2(0,-1)` while player is inside. The scrolling direction flips to y-axis scrolling.\n\n### Mode C — Zero-G Drift\n- ZeroGDriver: no gravity. Player has a persistent `velocity: Vec2` component\n- Gumball cannon: each fire adds `impulse = Vec2(-thrustMag * cos(facing), -thrustMag * sin(facing))` to velocity\n- Each tick: `player.pos += velocity; velocity *= (1 - dragCoefficient)`\n- Ammo = gumballs (consumable). At ammo=0: `scene.trigger('outOfAmmo')` → respawn at ship\n- Asteroids: `AsteroidEntity extends Entity`. `willDie()` spawns 0-2 smaller asteroids (configurable via `splitCount` and `splitSizeMultiplier`). Max 2 generations (size `L → M → S`).\n- Split algorithm: `spawnPos = parent.pos + Vec2(±splitRadius, random)`, velocity inherits parent velocity ± randomSpin\n\n### Mode D — Ship Broadside + Boarding\n- ShipBroadsideDriver: no walking. Timing mini-game: both ships exchange cannonball `Fireball` entities on interval. Hull HP = ship entity HP. Player-side: UI shows reload timer, player clicks to fire.\n- Boarding trigger: ship HP below threshold OR player uses 'board ship' ability → `scene.switchDriver(new HorizontalDriver())` + spawn player entity at ship-deck Y position + spawn crew enemies\n- All mid-quest mode switches share the same entity array; only the PhysicsDriver and player movement config change.\n\n---\n\n## 14.
