> For Candy Box 3's locked stack (vanilla TS + Vite + hand-rolled signals), the single most consequential finding is that CB3 does NOT need a big-number library. The design doc's peak is ~10^12 (Act 3 dyson scaffold), which sits comfortably inside the native f64 exact-integer range (Number.MAX_SAFE_INTEGER ≈ 9.007e15). break_infinity.js only earns its keep past ~1e308, ~6 orders of magnitude beyond CB3's ceiling; adopting it now would add a dependency, force a Decimal type through every counter and the signals layer, and break the cheap arithmetic the rest of these recommendations rely on. Use native number, keep all currencies as integers, and centralize formatting in one module so a swap stays cheap if scope ever explodes. For the loop, use a single requestAnimationFrame driver with a fixed-timestep accumulator (10 Hz sim tick, render every frame), clamped against the spiral of death; offline/background progress is a separate timestamp-delta catch-up path, not the rAF loop. For state, a ~40-line hand-rolled signal/computed/effect system with automatic dependency tracking (modeled on potch/signals but hardened) beats nanostores and zustand-vanilla for hundreds of small counters. Saves: a versioned envelope in localStorage with an ordered migration chain, LZ-string-compressed base64 export strings, validate-and-migrate on import, autosave on a debounced interval plus visibilitychange.

### Key findings
- CB3's wealth curve peaks at ~10^12 candies (DESIGN.md section 5: 'Act 3 ends ~10^12'). Native JS numbers represent every integer exactly up to Number.MAX_SAFE_INTEGER = 9,007,199,254,740,991 (~9.007e15) and carry usable float precision out to ~1.8e308. CB3's ceiling is ~6,000x below the safe-integer limit and ~10^296 below the float limit.
- break_infinity.js is purpose-built for values BIGGER than 1e308 (up to 1e9e15) and explicitly trades accuracy for speed; its whole reason to exist (a {mantissa, exponent} pair replacing the float exponent) provides zero benefit until you exceed 1e308. Its measured wins over decimal.js are on transcendental ops (pow ~442x, exp ~401x, log10 ~121x) that CB3 barely uses; add/mul are only ~2.5x and still far slower than a native + or *.
- decimal.js and OmegaNum.js/break_eternity.js are even more overkill: decimal.js prioritizes arbitrary-precision accuracy (slow), and break_eternity/OmegaNum target googological scales (10^^3 and beyond) that CB3 will never approach. None are justified by the design doc.
- The canonical idle game loop is one requestAnimationFrame driver with a fixed-timestep accumulator: accumulate frame delta into a 'lag' variable, run sim updates in a while-loop in fixed steps, render once per frame with an interpolation alpha = lag/step. A maxUpdates panic threshold prevents the 'spiral of death' when a backgrounded tab returns with hours of accumulated delta.
- Background tabs are throttled by browsers (rAF effectively pauses, timers clamp to ~1/sec or slower), so wall-clock time advances while the sim does not. The robust pattern is to treat foreground rAF and offline/background catch-up as two distinct code paths: the rAF loop runs the live sim; a timestamp-delta computation (now - lastSeenTimestamp) reconciles elapsed real time on focus/load.
- Offline progress is computed from saved timestamps, not from the running loop: persist lastTick wall-clock time, and on resume award income = rate * min(elapsed, cap). This is the same mechanism that powers CB3's 'candy box closed during quests = 2x accumulation' Schrodinger mechanic, which is just an offline-rate multiplier keyed on whether the box is closed.
- A complete automatic-dependency-tracking signal system fits in ~40-60 lines: a context stack records which effect/computed is currently running, signal getters register the running context as a dependent, signal setters notify dependents, and a batch() defers notifications. The potch/signals reference implementation demonstrates this but is explicitly a learning toy (no stale-dependency cleanup, recomputes computeds eagerly, no glitch-freedom) and must be hardened for CB3.
- nanostores is tiny (~265 bytes core) but its computed() requires you to declare each dependency explicitly (awkward DX for hundreds of interdependent upgrades); zustand-vanilla is a single-store/selector model better suited to React app state than to a graph of hundreds of independent reactive counters. Both add a dependency the locked stack doesn't need.
- Save best practice for web/idle games: localStorage is fine under ~1MB (CB3 will be far smaller); use a versioned save envelope {version, timestamp, state, checksum}; autosave every few minutes via setInterval PLUS on visibilitychange (more reliable than beforeunload, especially on mobile); export/import as compressed base64 strings (LZ-string), and always validate + migrate inside a try/catch on import.
- Number formatting for an ASCII game is solved natively: Intl.NumberFormat with notation:'compact' yields '988B'-style suffixes, and notation:'scientific' yields exponent form. For CB3's <=10^12 range, a tiny hand-rolled K/M/B/T suffix formatter is enough and gives full control over the deadpan tone (e.g. the '1,000,000,000 candies. The number has stopped meaning anything to you' beat needs comma grouping, not suffixes).

### Patterns to steal
- Single rAF driver + fixed-timestep accumulator with a `maxUpdates` panic break (from the HipHopHuman idle game-loop gist) - deterministic sim, smooth render, spiral-of-death-proof.
- Derive offline gains from a saved wall-clock timestamp delta rather than from the running loop (geekextreme delta-time article; Melvor Idle offline progression) - and run catch-up before starting the live loop.
- Context-stack automatic dependency tracking for signals (potch/signals): getters self-register the running observer, setters notify, computeds/effects push themselves as the active observer - ~40 lines, no framework.
- batch() to coalesce many writes into one notification pass (potch/signals) - essential when the offline catch-up updates dozens of counters at once.
- Versioned save envelope + ordered migration chain (bugnet save-best-practices; terraformental/itch save-migration devlogs): every save carries a schema version, migrations run 1->N sequentially on load.
- Autosave on visibilitychange (hidden) instead of relying on beforeunload - markedly more reliable on mobile (bugnet).
- LZ-string compressToBase64 for compact shareable save strings (multiple itch.io idle devs use this for export/import).
- Centralize Number.MAX_SAFE_INTEGER awareness and formatting in one module so a future big-number migration is a single-file change (lesson distilled from break_infinity's own 'only past 1e308' guidance).
- Intl.NumberFormat({notation:'compact'}) for instant locale-aware K/M/B/T suffixes when you want them, with no library.

### Pitfalls
- Do NOT adopt break_infinity.js / decimal.js 'to be safe.' It forces a Decimal object type through every counter, every comparison (no native <, must use .lt()), the signals layer, and save serialization - poisoning the cheap native arithmetic everything else here assumes, for numbers CB3 never reaches. It's a one-way door that's hard to back out of.
- The potch/signals reference is explicitly a learning toy and is unsafe as-is: (1) it never cleans up stale dependencies, so an effect that reads different signals on different runs keeps accumulating phantom subscriptions; (2) computeds recompute eagerly via an effect rather than lazily/memoized; (3) no equality check, so writing the same value re-fires effects. Harden all three before relying on it for hundreds of counters.
- Letting candies/sec be a float and accumulating per-frame instead of per-fixed-tick causes nondeterministic drift across machines with different refresh rates - apply production only inside the fixed-step loop using rate*(step/1000).
- Backgrounded tabs throttle rAF and timers (down to ~1 update/sec or fully paused), so a loop that 'just keeps ticking' will silently under-count while hidden. Relying on the live loop for background progress instead of a timestamp-delta catch-up will undercount idle income - exactly the income the Schrodinger box doubles.
- Not clamping offline elapsed time lets a clock change or a months-closed tab dump an absurd candy windfall (and can overflow even native numbers if combined with a high rate). Always clamp elapsed to a design cap.
- beforeunload is unreliable (especially mobile/iOS) for the final save - if it's your only save trigger you'll lose progress. Use visibilitychange + interval as primary; beforeunload only as a bonus.
- Surfacing the box 2x mechanic in any UI text breaks the design intent (section 18.12: discoverable, never stated). Keep it out of logs, tooltips, and the candies/sec breakdown.
- Storing signal wrapper objects directly in the save (instead of a plain serializable state object) will serialize getters/closures and break JSON.stringify - keep a plain state object as the source of truth and mirror it into signals.
- Trusting imported save strings without revalidating shape after migration will crash the game on hand-edited or version-skewed saves; always validate post-migration and reject gracefully.
- Intl.NumberFormat compact display is locale-dependent and can produce non-deadpan output (e.g. localized suffixes); for the tone-critical 'the number has stopped meaning anything to you' moments use explicit comma grouping, and pin the locale ('en-US') so output is stable.

### Recommendations
- LARGE NUMBERS: Use native `number` for all of CB3. Store every currency as an integer (candies, lollipops, stardust, etc.); never let a currency become a float. Do NOT add break_infinity.js, decimal.js, OmegaNum, or break_eternity. Adopt a big-number lib only if a future design change pushes any tracked value past ~1e15 (the safe-integer limit) - which the current DESIGN.md never does. To keep that escape hatch cheap, route ALL arithmetic-display through one module and consider a thin `Currency` alias so a swap is mechanical.
- FORMATTING: Hand-roll a ~30-line formatter module (format.ts) with two functions: `formatCount(n)` using `Intl.NumberFormat('en-US')` for full comma-grouped numbers (needed for the deadpan '1,000,000,000 candies' jokes), and `formatCompact(n)` for K/M/B/T suffixes on dense UI. Have it accept a `number` today; if you ever migrate to a big-number type, only this module changes. Use `notation:'scientific'` only if a value would otherwise overflow the suffix table.
- TICK LOOP: One `requestAnimationFrame` driver. Run the simulation at a FIXED 10 Hz tick (step = 100ms) via an accumulator; render every animation frame. Compute per-tick production as `rate_per_second * (step/1000)` and apply it once per fixed step so the sim is deterministic regardless of display refresh rate. Clamp the per-frame delta (e.g. max 250ms) and cap updates-per-frame (maxUpdates) so a refocused tab can't trigger a spiral of death. Keep render (DOM/ASCII text updates) strictly separate from the sim update.
- DECOUPLE SIM FROM RENDER: The sim tick mutates state (or pushes into signals); the render pass reads signals/computeds and writes DOM. Because CB3 is ASCII text, you generally do NOT need per-frame interpolation - update text on signal change, not every frame. Drive the per-tick economy through signals so the UI auto-updates; only the live counter (candies/sec ticking visibly) needs a per-frame refresh, and that can be a single effect.
- OFFLINE / BACKGROUND PROGRESS: Treat it as a separate catch-up path, not the rAF loop. Persist `lastTickWallClock = Date.now()` in the save and on every autosave. On load (and on `visibilitychange` -> visible), compute `elapsedMs = Date.now() - lastTickWallClock`, clamp to a sane cap (e.g. 24-72h for design intent, not infinity), and award `rate * elapsedSeconds` in one batched update. Run this BEFORE starting the rAF loop so the live loop never has to swallow a huge delta.
- SCHRODINGER BOX (2x while closed): Model the candy box as a boolean `boxClosed` flag in state. The offline/background catch-up multiplies idle income by 2 for the portion of elapsed time the box was closed (i.e. while the player was off on a quest with the box shut). Implement as `idleIncome = rate * elapsed * (boxClosed ? 2 : 1)`. Per DESIGN.md section 18.12 this is discoverable and never stated - so do NOT surface the multiplier in any tooltip or log line; only the resulting candy count reveals it.
- REACTIVE STATE: Hand-roll a minimal signals module (signal/computed/effect/batch) with automatic dependency tracking - do NOT pull in nanostores or zustand-vanilla. Use the context-stack pattern: a global `activeObserver`, signal getters subscribe the active observer, setters notify subscribers, effects/computeds set themselves as the active observer while running. Add the three things the potch reference toy lacks (see pitfalls): dependency cleanup on re-run, lazy/memoized computeds, and equality-gating so identical writes don't fire effects.
- STATE SHAPE: Make each independent counter/upgrade its own small signal (candies, candiesPerSecond, each upgrade level), and derive aggregates (total candy/sec, total cost, can-afford flags) as `computed`. This scales cleanly to hundreds of upgrades because each effect only re-runs when its specific dependencies change - which is exactly why fine-grained signals beat a single monolithic store here. Keep the canonical game state as a plain serializable object; mirror primitives into signals for reactivity (saving serializes the plain object, not the signal wrappers).
- SAVE: localStorage with a versioned envelope `{ v: <schemaVersion>, t: <savedAt>, lastTick: <wallClock>, state: {...} }`. Maintain an ordered migration chain `migrations[fromVersion](state) -> state` applied sequentially on load until current. Autosave on a debounced 30-60s interval AND on `visibilitychange`->hidden (more reliable than beforeunload, per the research). Keep `beforeunload` as a best-effort last write only.
- EXPORT/IMPORT STRINGS (series tradition): Serialize the envelope to JSON, compress with LZ-string (`compressToBase64`), and present that as the shareable save string. On import: try/catch around decompress+JSON.parse, run the SAME migration chain, then VALIDATE the resulting shape (a small zod schema or hand-written guards) before committing it - reject and keep the current save on any failure. Track `lifetimeCandiesEaten` as its own persisted field (DESIGN.md section 20) since it gates ending 3 and scales 'wrapper'; never derive it from balance.
- VALIDATION & ANTI-CHEAT POSTURE: Since this is single-player and saves are meant to be shared/edited (series tradition), do not fight tampering hard - just validate structure to avoid crashes. Optionally add a non-cryptographic checksum to the envelope to detect accidental corruption (show a 'this save looks corrupted' message) rather than to prevent cheating. Treat any field as untrusted on import: clamp to type, default missing fields via migrations.

## CB3 Technical Foundations Brief\n\nContext: peak tracked value per DESIGN.md section 5 is ~10^12 (Act 3 dyson scaffold), with candy/sec topping out around 1M+/s in Act 3. Stack is locked: vanilla TS + Vite + hand-rolled signals.\n\n---\n\n### 1. LARGE NUMBERS - Recommendation: native `number`, integers only. Do not add a library.\n\nThe decision is not close. Native JS `number` (IEEE-754 f64) represents **every integer exactly up to `Number.MAX_SAFE_INTEGER` = 9,007,199,254,740,991 (~9.007e15)** and retains usable precision out to ~1.8e308. CB3's ceiling of ~10^12 is about **6,000x below the exact-integer limit**. There is no precision problem to solve.\n\nWhat the libraries are actually for:\n- **break_infinity.js**: stores `{mantissa, exponent}` to represent values *bigger than 1e308* (up to 1e9e15). Its benefit begins precisely where the native float exponent runs out - 1e308 - which is ~10^296 past CB3's peak. Its headline speed wins are on transcendentals (pow ~442x, exp ~401x, log10 ~121x vs decimal.js) that CB3 barely touches; add/mul are only ~2.5x faster than decimal.js and *much slower* than a native `+`/`*`.\n- **decimal.js**: arbitrary precision, accuracy over speed - the opposite of what an idle game wants, and still unnecessary below 1e15.\n- **OmegaNum.js / break_eternity.js**: googological scales (10^^3+). Absurdly overkill.\n\nCost of adopting one anyway: a `Decimal` object type infects every counter, every comparison (`a < b` becomes `a.lt(b)`), the signals layer, and save serialization. It's a one-way door.\n\n**Concrete rule for when to adopt:** only if a future design change pushes any *tracked, persisted* value past ~1e15. The current doc never does. To keep the option cheap, funnel all arithmetic/display through a single module and use a `type Currency = number` alias now.\n\n**Formatting** (a real CB3 requirement, section 2 jokes depend on it): hand-roll `format.ts`.\n```ts\nconst grouped = new Intl.NumberFormat('en-US') // pinned locale for stable tone\nexport const formatCount = (n: number) => grouped.format(Math.floor(n))\nconst SUFFIX = ['', 'K', 'M', 'B', 'T']\nexport function formatCompact(n: number): string {\n  if (n < 1000) return String(Math.floor(n))\n  const tier = Math.min(Math.floor(Math.log10(n) / 3), SUFFIX.length - 1)\n  const scaled = n / 10 ** (tier * 3)\n  return scaled.toFixed(scaled < 10 ? 2 : scaled < 100 ? 1 : 0) + SUFFIX[tier]\n}\n```\nUse `formatCount` for the deadpan beats ('You have 1,000,000,000 candies...'), `formatCompact` for dense UI. `Intl.NumberFormat(.., {notation:'compact'})` is a zero-dep alternative but its localized suffixes are less tone-controllable.\n\n---\n\n### 2. GAME / TICK LOOP - Recommendation: one rAF driver + fixed-timestep accumulator at 10 Hz sim, render per frame.\n\nWhy not bare `setInterval`: drifts, fires in background-throttled bursts, not aligned to paint. Why not pure variable-dt rAF: nondeterministic economy across refresh rates. The accumulator gives a deterministic sim plus smooth render.\n```ts\nconst STEP_MS = 100              // 10 Hz sim\nconst MAX_DELTA = 250            // clamp a refocus spike\nconst MAX_UPDATES = 50           // spiral-of-death break\nlet last = performance.now(), lag = 0\n\nfunction frame(now: number) {\n  lag += Math.min(now - last, MAX_DELTA)\n  last = now\n  let updates = 0\n  while (lag >= STEP_MS) {\n    simTick(STEP_MS / 1000)      // dt in seconds\n    lag -= STEP_MS\n    if (++updates >= MAX_UPDATES) { lag = 0; break }\n  }\n  render()                       // reads signals -> ASCII/DOM\n  requestAnimationFrame(frame)\n}\nrequestAnimationFrame(frame)\n```\nPer-tick production: `candies += candiesPerSecond * dt`. Because CB3 is ASCII text, you rarely need per-frame interpolation - update text reactively on signal change, with one effect for the visibly-ticking live counter. Keep `simTick` (state mutation) and `render` (DOM writes) strictly separate.\n\n---\n\n### 3. OFFLINE / BACKGROUND PROGRESS - Recommendation: separate timestamp-delta catch-up, run before the live loop.\n\nBackground tabs throttle rAF/timers (down to ~1/s or paused), so the live loop under-counts while hidden. Do not rely on it for background gains. Instead persist wall-clock and reconcile on resume:\n```ts\nfunction catchUp(state: GameState, nowMs = Date.now()): GameState {\n  const elapsedS = Math.min((nowMs - state.lastTickWallClock) / 1000, OFFLINE_CAP_S)\n  const mult = state.boxClosed ? 2 : 1      // Schrodinger box, section 18.12\n  const gained = state.candiesPerSecond * elapsedS * mult\n  return { ...state, candies: state.candies + gained, lastTickWallClock: nowMs }\n}\n```\nCall on load and on `visibilitychange`->visible, BEFORE starting rAF, so the live loop never swallows a huge delta. `OFFLINE_CAP_S` is a design knob (start ~24-72h). The box-closed 2x is just `mult`; per the doc it must never appear in any UI/log - only the candy total reveals it.\n\n---\n\n### 4. REACTIVE STATE - Recommendation: ~40-60 line hand-rolled signals with auto dep-tracking. Not nanostores, not zustand-vanilla.\n\nFor hundreds of small interdependent counters/upgrades, fine-grained signals win: each `effect` re-runs only when its specific dependencies change. nanostores' `computed` makes you declare each dependency manually (awkward at this scale); zustand-vanilla's single-store/selector model is React-app shaped, not a counter-graph.\n\nThe potch/signals context-stack pattern is the right skeleton but is a known toy - it lacks (a) stale-dependency cleanup, (b) lazy/memoized computeds, (c) equality gating. Harden those:\n```ts\ntype Sub = { run(): void; deps: Set<Set<Sub>> }\nlet active: Sub | null = null\nconst batched = new Set<Sub>(); let batching = false\n\nexport function signal<T>(initial: T) {\n  let value = initial\n  const subs = new Set<Sub>()\n  return {\n    get(): T { if (active) { subs.add(active); active.deps.add(subs) } return value },\n    set(next: T) {\n      if (Object.is(next, value)) return            // (c) equality gate\n      value = next\n      const list = [...subs]\n      for (const s of list) batching ? batched.add(s) : s.run()\n    },\n  }\n}\n\nexport function effect(fn: () => void) {\n  const sub: Sub = {\n    deps: new Set(),\n    run() {\n      for (const d of sub.deps) d.delete(sub)        // (a) cleanup before re-track\n      sub.deps.clear()\n      const prev = active; active = sub\n      try { fn() } finally { active = prev }\n    },\n  }\n  sub.run()\n  return () => { for (const d of sub.deps) d.delete(sub); sub.deps.clear() }\n}\n\nexport function computed<T>(fn: () => T) {        // (b) memoized via cached signal\n  const s = signal<T>(undefined as unknown as T)\n  effect(() => s.set(fn()))\n  return { get: s.get }\n}\n\nexport function batch(fn: () => void) {\n  batching = true; try { fn() } finally { batching = false }\n  const run = [...batched]; batched.clear()\n  for (const s of run) s.run()\n}\n```\nState shape: each independent counter/upgrade level is its own signal; aggregates (total c/s, total cost, canAfford) are `computed`. Keep the canonical, serializable game state as a plain object and mirror primitives into signals - save serializes the plain object, not signal wrappers/closures. Use `batch()` around the offline catch-up (it touches many counters at once).\n\n---\n\n### 5. SAVE - Recommendation: localStorage versioned envelope + ordered migrations; LZ-string base64 export; validate-and-migrate on import; debounced interval + visibilitychange autosave.\n\nCB3 saves are tiny (well under 1MB), so localStorage is correct; no IndexedDB needed.\n```ts\ninterface SaveEnvelope { v: number; t: number; lastTick: number; state: GameState }\nconst CURRENT_VERSION = 3\nconst migrations: Record<number, (s: any) => any> = {\n  1: s => ({ ...s, lifetimeCandiesEaten: s.lifetimeCandiesEaten ?? 0 }),\n  2: s => ({ ...s, boxClosed: false }),\n  // ...applied sequentially fromVersion -> CURRENT_VERSION\n}\nfunction migrate(env: any): SaveEnvelope {\n  let { v, state } = env\n  while (v < CURRENT_VERSION) { state = migrations[v + 1](state); v++ }\n  return { ...env, v: CURRENT_VERSION, state }\n}\n```\n- **Autosave**: debounced `setInterval` ~30-60s, PLUS `visibilitychange`->hidden (more reliable than beforeunload, esp. mobile). `beforeunload` only as best-effort.\n- **Export/import** (series tradition): `LZString.compressToBase64(JSON.stringify(env))` -> shareable string. On import: try/catch around decompress + `JSON.parse`, run the SAME `migrate()`, then VALIDATE shape (small zod schema or guards) before committing; on any failure keep the current save and show a friendly message.\n- **lifetimeCandiesEaten** is a first-class persisted field (gates ending 3, scales 'wrapper', section 20) - never derive it from balance.\n- **Anti-cheat posture**: single-player, saves meant to be shared/edited - don't fight tampering. Add a non-cryptographic checksum only to flag accidental corruption. Treat every imported field as untrusted: clamp types, default via migrations.\n\n---\n\n### Phase-1 mapping (DESIGN.md section 21)\nThe vertical slice needs exactly: native-number economy + `format.ts`, the rAF accumulator loop, the hardened signals module, the localStorage envelope + export string, and the offline catch-up (the box 2x can land in Phase 1 since it's just a multiplier).
